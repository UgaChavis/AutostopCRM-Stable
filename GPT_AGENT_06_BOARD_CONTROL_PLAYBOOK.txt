AUTOSTOP CRM BOARD CONTROL PLAYBOOK

Purpose
- This file tells the agent how to safely read and manage the current board through MCP.
- It is focused on practical board control, not server deployment.

Absolute operating rule
- The current AutoStop CRM board is the source of truth.
- Work only through the current MCP connector and the board API behind it.
- Do not mix this board with any other kanban, CRM, or sandbox board.

Default startup sequence
1. ping_connector
2. bootstrap_context
3. get_runtime_status if there is any doubt about MCP or API health
4. review_board or get_gpt_wall for full context

How to choose the right read tool
- review_board: quick operational summary across the whole board
- get_gpt_wall: broad hidden context, recent changes, and card narratives
- get_board_snapshot: precise machine-readable state after writes
- search_cards: fast card lookup by text, vehicle, problem, or id fragment
- get_card_context: best first read when the task is clearly about one card
- get_card: full card payload before and after edits
- get_card_log: timeline when history matters

How to choose the right write tool
- create_card: new card only when the user really wants a new case
- update_card: edit existing title, description, vehicle, tags, notes, and metadata
- move_card: single-card move
- bulk_move_cards: group move between columns
- archive_card: archive only when there is no open repair order
- restore_card: return archived card to work
- set_card_deadline: explicit deadline management
- set_card_indicator: visual signal or urgency mark

Board management rules
- Read before write.
- After each important write, do read-after-write.
- Prefer a single clean update_card over multiple tiny writes when editing one card.
- Prefer bulk_move_cards over loops of move_card for group moves.
- Keep user-entered facts authoritative.
- If data is ambiguous, preserve uncertainty instead of inventing details.

Card creation rules
- A new card may exist without a repair order.
- Do not assume a repair order exists on every card.
- When creating a card, focus on vehicle, title, description, tags, deadline, and indicator.

Archive and restore rules
- A card without a repair order may be archived.
- A card with a closed repair order may be archived.
- A card with an open repair order must not be archived.
- If an old inconsistent archived + open repair order state is encountered, surface a controlled error and avoid silent destructive changes.

Recommended safe sequence for a single card task
1. search_cards or get_card_context
2. get_card
3. update_card or move_card
4. get_card
5. get_board_snapshot if the move affected board state materially
