AUTOSTOP CRM SERVER AGENT WORKFLOW

Updated on: 2026-04-13

Core runtime model
1. A task is added to the queue manually, by schedule, by on_create trigger, or by card autofill.
2. The server agent picks the oldest pending task.
3. The model works only through the current AutoStop CRM tool surface.
4. Tool calls append structured results to the run context.
5. The agent returns a concise result and writes action/run logs.

Rules for external tools
- Use only whitelisted domains through the built-in adapters.
- Keep external lookups tight and practical.
- Stay within the per-run request limit.
- First search, then consolidate, then write concise structured value back into the card.

Scheduled tasks
- once: single queued run
- interval: repeated run with minute or hour interval
- on_create: fires when a new card appears in the selected scope

Task scopes
- current_card
- column
- all_cards

Duplicate-run protection
- One active run per scheduled task id.
- One active on_create run per schedule + card pair.
- One active card autofill run per card.

Card autofill flow
1. User enables card autofill in the Agent window.
2. The card stores ai_autofill_active and ai_autofill_until.
3. The first pass is enqueued immediately.
4. Follow-up checks continue for up to 4 hours.
5. If the card changed, the next pass is scheduled sooner.
6. If the card is quiet or waiting, the next pass is delayed.
7. The cycle stops after timeout or when the mode is disabled.

Card autofill output style
- Read the full card first.
- Keep existing user text authoritative.
- Add VIN, parts, ТО, DTC, and fault hints only when supported by the card data.
- Write short, structured additions instead of long walls of text.
- Prefer incremental updates over full rewrites.

Repair-order business rules
- A card may exist without a repair order.
- A repair order is created lazily on first open from the card.
- An open repair order blocks card archive.
- An unpaid repair order must not be closed.

Read-after-write discipline
- After create_card, update_card, archive_card, restore_card, update_repair_order, payment actions, or card autofill, read the affected entity again.
- Use get_card, get_card_context, get_repair_order, get_board_snapshot, or get_gpt_wall depending on scope.
