AUTOSTOP CRM TASKS, AUTOFILL, AND INTERNET TOOLS

Purpose
- This file explains scheduled tasks, on_create automation, card autofill, and the limited internet tools available to the server agent.

Scheduled task model
- Tasks are stored as lightweight metadata.
- Scheduler runs inside the current server agent process.
- No cron dependency is required.
- The scheduler periodically checks due tasks and enqueues them.

Supported schedule types
- once
- interval
- on_create

Supported task scopes
- current_card
- column
- all_cards

Task creation rules
- current_card: use when the task is created from the currently open card.
- column: use when the task should react only to one column.
- all_cards: use for board-wide monitoring or review.

on_create trigger behavior
- Fires when a new card appears.
- May be limited to one selected column.
- Should be used for inbox enrichment, first-pass classification, or initial cleanup.
- Must not enqueue a duplicate active run for the same schedule and card.

Card autofill mode
- UI button enables autofill for 4 hours on the current card.
- The first run is immediate.
- Follow-up runs are adaptive, not constant.
- Stored metadata includes:
  - ai_autofill_active
  - ai_autofill_until
  - ai_next_run_at
  - last_ai_run_at
  - ai_run_count
  - last_card_fingerprint

Adaptive follow-up rules
- If the card changed after the last AI pass, schedule the next pass sooner.
- If there are no meaningful changes, slow down checks.
- If the text implies waiting, pause, parts on the way, or client decision, slow down more aggressively.
- Keep the number of runs bounded.
- Never run parallel autofill passes for the same card.

Available internet tools
- decode_vin(vin)
- find_part_numbers(query, vehicle)
- estimate_price_ru(part_number, vehicle)
- decode_dtc(code)
- search_fault_info(query)

Internet-tool rules
- Use the built-in adapters only.
- Keep lookups practical and limited.
- Prefer exact identifiers from the card first.
- Return compact structured results, then write concise value into the card.

Recommended autofill strategy
1. Read the card fully.
2. Build compact context:
   - card fields
   - description
   - vehicle and vehicle_profile
   - previous AI notes
   - AI log tail
   - recent events
   - short board context when VIN or stable vehicle context justifies it
3. Detect whether there is a VIN, DTC, part request, maintenance request, mileage, or symptom description.
4. Select scenarios before any writes:
   - VIN enrichment
   - parts lookup
   - maintenance lookup
   - DTC lookup
   - fault research
   - normalization
5. Use only the tools that actually match the selected scenarios.
6. Consolidate findings into short Russian operational notes.
7. Write short structured additions without overwriting good manual data.

Current orchestration rules
- VIN is always attempted first when VIN exists.
- Parts lookup uses vehicle context and query aliases for common Russian part names.
- Maintenance lookup uses mileage and known capacities when they are present.
- Fault research should stay compact and practical.
- Result writing must be additive: preserve original notes, prices, article numbers, phone numbers, and operator comments.
- If the card already contains an AI note with the same meaning, do not duplicate it.
- If certainty is low, add one short follow-up note for the next operator instead of fabricating facts.

Current external tool routing
- decode_vin(vin)
  First priority when VIN is present. Use its result to patch missing vehicle fields and to improve later searches.
- find_part_numbers(query, vehicle)
  Use for concrete parts such as radiator, control arm, strut, bearing, brake pads, thermostat, pump, belt, chain, filters, plugs, battery.
- estimate_price_ru(part_number, vehicle)
  Use only when there is a plausible OEM or analog number.
- decode_dtc(code)
  Use for explicit DTCs. Add short meaning, likely causes, and first checks.
- search_fault_info(query)
  Use for symptom-only cards when there is no precise part or code yet.
